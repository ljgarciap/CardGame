# Architecture Document - Card Game

## Overview
This document outlines the technical architecture for the multiplatform card game.

## Technology Stack
- **Frontend**: Flutter (Targeting Android, iOS, and Web).
- **Backend**: Python (FastAPI).
- **Database**: PostgreSQL.
- **Communication**: Real-time via WebSockets.

## Key Components
1. **Game Engine**: Handles logic, turns, and state validation.
2. **Gacha Engine**: Manages pack opening, probability calculations, and card generation.
3. **Real-time Server**: Manages concurrent player connections and message broadcasting.
4. **Matchmaking**: Logic to pair players or create private rooms.

## Backend Conventions

### Alembic migrations with Postgres ENUM columns
This bug class has already bitten the gacha-engine migrations **twice**
(`docs/reviews/gacha-engine.md`) before being caught by manual verification.
Checklist for any migration that adds an `Enum`-typed column:

1. **Reusing an existing ENUM type** (e.g. a second table also using
   `rank`/`rarity`/`faction`): `alembic revision --autogenerate` emits a
   plain `sa.Enum(..., name='rank')`, which tries to `CREATE TYPE` again and
   fails with `DuplicateObject` on a clean upgrade — even though the table
   is new, the *type* isn't. Fix it by hand in the generated migration:
   ```python
   from sqlalchemy.dialects import postgresql
   _rank_enum = postgresql.ENUM(
       'hero', 'demigod', 'minor_god', 'major_god', name='rank', create_type=False
   )
   # use _rank_enum instead of sa.Enum(...) in every column of this migration
   # that reuses the type
   ```
2. **Downgrading a migration that created a new ENUM type**: the
   autogenerated `downgrade()` only has `op.drop_table(...)` calls — it
   never drops the ENUM type the table's column created. That leaves an
   orphaned type in Postgres, which breaks a *later* re-upgrade with the
   same `DuplicateObject` error. Fix it by hand:
   ```python
   def downgrade() -> None:
       op.drop_table('card_archetypes')
       # ... other drop_table calls ...
       sa.Enum(name='rank').drop(op.get_bind(), checkfirst=True)
       sa.Enum(name='rarity').drop(op.get_bind(), checkfirst=True)
   ```
   Only drop a type here if **no other table still using it** survives the
   downgrade — if another table in an earlier migration also uses `rank`,
   dropping the type would break that table's rows.
3. **Always verify with a real Postgres, not just `Base.metadata.create_all`**
   (which is what the test suite's `conftest.py` uses, and it doesn't
   exercise Alembic at all): run `alembic upgrade head` →
   `alembic downgrade -1` → `alembic upgrade head` against a real Postgres
   container before considering a new migration done. Both bugs above only
   surface on that third step, not on a plain first `upgrade`.
4. **`ADD COLUMN ... NOT NULL` on an existing table**: needs an explicit
   `server_default=...` in the migration (autogenerate doesn't add one) or
   it breaks against any table that already has rows — even if your local
   dev DB happens to be empty when you write the migration.
